# Agent Guide — __PROJECT_NAME__

## Project identity

`__PROJECT_NAME__` is a PostgreSQL-first TypeScript application generated by `create-agentic-sql-app`.

The sample domain demonstrates project membership, release items, dashboard reads, and release transitions. All business behavior is owned by PostgreSQL stored procedures.

## Core rules

1. **Business logic belongs in PostgreSQL.**
2. **Fastify API routes are thin.**
3. **React renders contract data.**
4. **The contract module is the source of truth for routes, errors, and response shapes.**

If a change violates one of these rules, stop and propose a correction plan before implementing it.

## Contract workflow

- Edit `contract/src/endpoints.ts` and `contract/src/errors.ts` before changing routes or generated docs.
- Run `npm run contract:generate` after contract changes.
- Do not hand-edit generated `DB_API_CONTRACT.md`, `ERROR_CODES.md`, or `contract/openapi.json`.
- Run `npm run agent:check` to verify contract freshness and drift rules.

## PostgreSQL responsibilities

PostgreSQL owns:

- permissions
- validations
- state transitions
- business metrics
- business-rule errors
- data integrity
- transactional behavior
- stored procedure contracts

When adding behavior, start with a stored procedure and database tests.

## API responsibilities

The API may:

- parse request params
- read actor context
- call stored procedures
- map database errors
- return JSON

The API must not:

- calculate business metrics
- validate business transitions
- implement permission rules
- duplicate database logic
- invent response fields

## React responsibilities

React may:

- render data
- call loaders
- call actions
- show loading, empty, and error states
- submit mutations

React must not:

- calculate business metrics
- infer permissions
- derive release status rules
- invent business fields
- duplicate database behavior

UI polish is allowed, but business rules must not move into React.

## Testing rules

- Run the required tests for the task type before marking work complete.
- Database changes require `npm run test:db`.
- API changes require `npm run test:server`.
- Frontend changes require `npm run test:web`.
- End-to-end flows require `npm run test:e2e` when a scenario is affected.
- Prefer adding or updating tests in the same change as the implementation.

## Stop conditions

Stop and summarize when:

- the requested task acceptance criteria are met
- the requested task scope is complete
- a contract, migration, or test failure blocks safe progress
- the requested change would move business logic out of PostgreSQL
- you need product decisions not documented in `TASKS.md`, `contract/src`, or generated contract docs

Do not continue into unrelated tasks without an explicit request.

## Anti-patterns

- Putting permission checks in Fastify routes or React components
- Computing release status or business metrics in TypeScript
- Returning fields not produced by a stored procedure
- Skipping database tests for procedure changes
- Modifying API or React files while implementing a database-only task
- Modifying database procedures while implementing a UI-only task without contract updates
- Inventing undocumented error codes

## Summary expectations

When you finish a task, summarize:

1. files changed
2. tests added or updated
3. commands run
4. assumptions made
5. recommended next task from `TASKS.md`
